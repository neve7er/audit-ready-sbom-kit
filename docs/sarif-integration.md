# SARIF Integration

> Last updated: 2026-05-02

## Overview

The `--format sarif` output option generates a SARIF 2.1.0 document from `TriageResult[]`, suitable for upload to GitHub Advanced Security via `actions/upload-sarif`. This enables vulnerability findings to appear in the GitHub Security tab alongside SAST and container scanning results.

## SARIF Version

**2.1.0** — `https://json.schemastore.org/sarif-2.1.0.json`

## Pipeline Position

```
PackageNode[] → applyTriage() → TriageResult[] → applyPolicy(.audit-policy.json) → PolicyResult[]
                                                                                        │
                                                                                        ▼
                                                                          SARIF serializer (--format sarif)
                                                                                        │
                                                                                        ▼
                                                                               results.sarif → actions/upload-sarif@v1
```

The SARIF adapter operates on `PolicyResult[]` — the fully resolved output after policy exceptions have been applied.

## Mapping: `ReasonCode` → SARIF `level`

| `reasonCode` | SARIF `level` | Rationale |
|---|---|---|
| `DIRECT_CRITICAL` | `error` | Directly depended critical — immediate action required |
| `DIRECT_HIGH` | `error` | Directly depended high severity |
| `TRANSITIVE_CRITICAL` | `warning` | Transitive critical — not directly controlled but reachable |
| `TRANSITIVE_HIGH` | `warning` | Transitive high |
| `DEV_DEPENDENCY` | `note` | Dev-only — not in production ship artifact |
| `OPTIONAL_DEPENDENCY` | `note` | Optional — not installed unless explicitly requested |
| `REACHABILITY_LOW` | `note` | Vulnerable code path is unreachable per reachability analysis |

The mapping table is defined in `src/adapters/output/sarif/levels.ts` and is the single source of truth. It must be updated if new `reasonCode` values are introduced.

## Required SARIF Fields

Every SARIF `result` object generated by this tool **must** contain:

| Field | Value | Source |
|-------|-------|--------|
| `ruleId` | `reasonCode` string (e.g. `"DIRECT_CRITICAL"`) | `PolicyResult.reasonCode` |
| `message.text` | The `reason` string from `.audit-policy.json` if this result is waived; otherwise the `rationale` from `TriageResult` | `PolicyResult` / `TriageResult` |
| `level` | See mapping table above | Derived from `reasonCode` via `levels.ts` |
| `locations[0].physicalLocation.artifactLocation.uri` | The dependency name from the lockfile entry | `PolicyResult.packageName` |
| `locations[0].physicalLocation.artifactLocation.uriBaseId` | `"ROOT"` — lockfile location is relative to the scanned project root | Static `"ROOT"` |

### Minimal SARIF Result Shape

```json
{
  "ruleId": "TRANSITIVE_CRITICAL",
  "level": "warning",
  "message": {
    "text": "Vulnerability CVE-2025-00001 is transitive and has reachability weight 0.5. Package lodash@4.17.20 is not directly depended."
  },
  "locations": [
    {
      "physicalLocation": {
        "artifactLocation": {
          "uri": "lodash",
          "uriBaseId": "ROOT"
        }
      }
    }
  ]
}
```

## Limitation: Source-Level Locations

**Physical file locations are approximated from lockfile entries — not source-level.**

The SARIF document identifies packages by their lockfile name only (e.g., `"lodash"`). It does **not** emit `lineNumber` or `region` data, because:
- The triage engine operates on dependency-graph data, not on source AST analysis
- Mapping from a package name to specific source files where it is imported requires a separate static-analysis pass

As a result, GitHub Advanced Security will show findings at the repository level but will not link them to specific source files or lines. This limitation is documented in the `compliance` section of the SARIF run metadata.

## GitHub Actions Upload

```yaml
- name: Run SBOM audit scan
  run: npx audit-ready scan --format sarif --dry-run --ignore-dev-deps > results.sarif

- name: Upload SARIF to GitHub Advanced Security
  uses: actions/upload-sarif@v3
  with:
    sarif_file: results.sarif
    category: "sbom-triage"
```

**Notes:**
- Use `--dry-run` in CI to skip OSV network calls (cache locally first; use `--force-refresh` in a nightly job)
- `category: "sbom-triage"` groups these findings separately from other code-scanning tools in the GitHub Security tab
- Upload does **not** block pull requests. Use branch protection rules to require that code-scanning alerts are resolved before merge

## Compliance Note

The SARIF run object must include a `compliance` notification when location data is absent:

```json
{
  "run": {
    "tool": {
      "driver": {
        "name": "audit-ready-sbom-kit",
        "version": "<version>",
        "informationUri": "https://github.com/example/audit-ready-sbom-kit",
        "rules": [...]
      }
    },
    "results": [...],
    "columnKind": "utf16CodeUnits",
    "properties": {
      "comment": "Location data is approximated from lockfile entries — not source-level. See docs/sarif-integration.md"
    }
  }
}
```
